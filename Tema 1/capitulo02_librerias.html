El Ecosistema de Bibliotecas de Python

Las bibliotecas de Python constituyen colecciones de código especializado que extienden las capacidades del lenguaje base. En lugar de implementar funcionalidades desde cero, los desarrolladores pueden utilizar bibliotecas desarrolladas y optimizadas por la comunidad.

El verdadero potencial de Python para ciencia de datos reside en su ecosistema de bibliotecas especializadas. El trabajo profesional en este campo raramente requiere implementaciones desde cero; en cambio, se emplean herramientas probadas y optimizadas por expertos.

1. Instalación de Python

Antes de trabajar con el ecosistema de bibliotecas de Python, es necesario tener una instalación correcta del intérprete de Python en el sistema. Esta sección detalla el proceso de instalación para los principales sistemas operativos.

📌 Versión Recomendada:

Se recomienda utilizar Python 3.10 o superior. Las versiones 3.11 y 3.12 ofrecen mejoras significativas de rendimiento (10-25% más rápidas) y características modernas. Evitar versiones anteriores a Python 3.9.

1.1 Verificar si Python está Instalado

Antes de proceder con la instalación, es conveniente verificar si Python ya está disponible en el sistema:

Comprobar instalación existente

# Windows (PowerShell o CMD)
python --version
python3 --version

# macOS / Linux
python3 --version

# Ver ubicación del ejecutable
# Windows:
where python
# macOS / Linux:
which python3

1.2 Instalación en Windows

Windows no incluye Python por defecto. Existen varios métodos de instalación:

Método 1: Instalador Oficial (Recomendado)

Pasos de instalación:
  1. Descargar instalador:
    • Visitar python.org/downloads
    • Descargar la última versión estable para Windows (archivo .exe)
    • Recomendado: Versión 3.12.x o 3.11.x
  2. Ejecutar instalador:
    • Doble clic en el archivo descargado
    • ⚠️ IMPORTANTE: Marcar casilla "Add Python to PATH"
    • Seleccionar "Install Now" (instalación estándar)
    • O "Customize installation" para opciones avanzadas
  3. Verificar instalación:
    • Abrir PowerShell o CMD
    • Ejecutar: python --version
    • Ejecutar: pip --version

Si al ejecutar python se abre Microsoft Store en lugar del intérprete:

  1. Abrir Configuración → Aplicaciones → Aplicaciones y características
  2. Buscar "Alias de ejecución de aplicaciones"
  3. Desactivar alias de python.exe y python3.exe

2. Gestión de Paquetes con pip y uv

Una vez instalado Python, el siguiente paso es comprender cómo instalar y gestionar las bibliotecas externas. Python ofrece principalmente dos herramientas modernas para esta tarea: pip (el estándar) y uv (la alternativa moderna y rápida).

2.1 pip: El Gestor de Paquetes Estándar

pip (Package Installer for Python) es el gestor de paquetes oficial, incluido por defecto desde Python 3.4. Es la herramienta más utilizada para instalar bibliotecas desde PyPI (Python Package Index).

⚠️ Importante - Instalación Global: En esta sección, los paquetes se instalan en el entorno Python global (sin entornos virtuales). Más adelante (Sección 3) aprenderás a crear entornos virtuales para aislar proyectos. Por ahora, solo estamos conociendo los comandos básicos.

Comandos básicos de pip

# Instalar un paquete
pip install numpy

# Instalar versión específica
pip install pandas==2.1.0

# Instalar múltiples paquetes
pip install numpy pandas matplotlib

# Actualizar un paquete
pip install --upgrade numpy

# Desinstalar un paquete
pip uninstall numpy

# Listar paquetes instalados
pip list

# Mostrar información de un paquete
pip show pandas

2.2 uv: El Gestor de Paquetes Moderno

uv es una herramienta moderna desarrollada por Astral (creadores de Ruff) que ofrece velocidad significativamente superior a pip (10-100x más rápido) y mejor gestión de dependencias.

Instalación de uv

Instalar uv según sistema operativo

# Windows (PowerShell o CMD)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# macOS y Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Verificar instalación
uv --version

Comandos básicos de uv

⚠️ IMPORTANTE: Dos formas de trabajar con uv
  • Proyectos con pyproject.toml (uv init): Usa uv add para gestión declarativa
  • Entornos simples (solo uv venv): Usa uv pip install como con pip tradicional
Más información sobre uv:

Consulta la documentación oficial de uv en https://docs.astral.sh/uv/ para detalles avanzados, opciones de instalación y referencias de comandos.

Entorno simple con uv venv (sin uv init ni pyproject.toml)

# 1. Crear entorno virtual simple
uv venv

# 2. Activar entorno
# Windows: .venv\Scripts\Activate.ps1
# macOS/Linux: source .venv/bin/activate

# 3. Instalar paquetes con uv pip
uv pip install numpy pandas matplotlib

# Instalar desde requirements.txt
uv pip install -r requirements.txt

# Listar paquetes instalados
uv pip list

# Desinstalar paquete
uv pip uninstall numpy

Proyectos con gestión declarativa (con uv init)

# 1. Inicializar proyecto (crea pyproject.toml)
uv init mi_proyecto
cd mi_proyecto

# 2. Crear entorno virtual
uv venv

# 2. Crear entorno virtual con una versión específica de Python
# uv venv --python=3.14

# 3. Activar entorno (necesrio si voy a ejecutar scripts desde línea de comandos)
# Windows: .venv\Scripts\Activate.ps1
# macOS/Linux: source .venv/bin/activate

# 4. ✅ RECOMENDADO: Agregar paquetes (registra en pyproject.toml)
# Si agrego paquetes con este comando no es necesario crear entorno virtual
uv add numpy pandas matplotlib

# Agregar versión específica
uv add "pandas>=2.1.0"

# Agregar dependencias de desarrollo
uv add --dev ipykernel

# Remover paquete del proyecto
uv remove numpy

# ⚠️ Evita usar uv pip install en proyectos con pyproject.toml
# (instalaría pero NO registraría la dependencia)

3. Entornos Virtuales

Los entornos virtuales son espacios aislados que permiten mantener dependencias de diferentes proyectos separadas. Esto evita conflictos entre versiones de bibliotecas y facilita la reproducibilidad.

3.1 ¿Por qué son Necesarios los Entornos Virtuales?

Imaginar la siguiente situación común en desarrollo profesional:

Escenario sin entornos virtuales (Problema)

  • Proyecto A (antiguo): Requiere Python 3.8, Pandas 1.3.0 y NumPy 1.20.0
  • Proyecto B (nuevo): Requiere Python 3.12, Pandas 2.1.0 y NumPy 1.26.0
  • Conflicto: No se pueden tener ambas versiones de Python y bibliotecas simultáneamente en el sistema
  • Resultado: Al actualizar para el Proyecto B, el Proyecto A deja de funcionar
🎯 Problemas que Resuelven los Entornos Virtuales:
  1. Diferentes Versiones de Python:
    • Proyecto antiguo requiere Python 3.8, proyecto nuevo usa Python 3.12
    • Sin entornos: solo puedes tener una versión de Python activa
    • Con entornos: cada proyecto usa su versión de Python
    • 🚀 uv lo hace súper fácil: uv venv --python 3.8 descarga e instala automáticamente Python 3.8 si no lo tienes
  2. Conflictos de Versiones de Bibliotecas:
    • Diferentes proyectos requieren diferentes versiones de la misma biblioteca
    • Sin entornos virtuales: imposible tener pandas 1.3 y pandas 2.1 simultáneamente
    • Con entornos virtuales: cada proyecto tiene su propia versión aislada
  3. Contaminación del Sistema:
    • Instalar bibliotecas en el Python del sistema puede romper herramientas del SO
    • En macOS/Linux, el sistema operativo usa Python para scripts internos
    • Modificar el Python del sistema puede causar fallos en el SO
  4. Dependencias Transitivas:
    • Una biblioteca puede requerir versiones específicas de otras (dependencias)
    • Proyecto A instala librería X que requiere NumPy 1.20
    • Proyecto B instala librería Y que requiere NumPy 1.26
    • Conflicto inevitable sin aislamiento
  5. Reproducibilidad:
    • Compartir proyecto con equipo o desplegar en servidor
    • Sin entorno: "funciona en mi máquina" pero no en otras
    • Con entorno: mismo comportamiento garantizado en todos lados
  6. Experimentación Segura:
    • Probar nuevas bibliotecas sin afectar proyectos existentes
    • Si algo sale mal: eliminar entorno y crear uno nuevo
    • No afecta otros proyectos ni el sistema
  7. Gestión de Proyectos Legacy:
    • Mantener proyectos antiguos funcionando
    • Proyecto del 2020 con Python 3.7 y Pandas 1.0
    • Nuevo proyecto con Python 3.12 y Pandas 2.1
    • Ambos funcionan perfectamente en sus entornos
📊 Ejemplo Práctico Real:

Supongamos trabajar en tres proyectos simultáneamente:

  • Proyecto "Análisis Ventas 2023" (Legacy): Python 3.8, pandas==1.5.3, matplotlib==3.6.0
  • Proyecto "ML Predicción": Python 3.11, pandas==2.1.0, scikit-learn==1.3.0, tensorflow==2.15.0
  • Proyecto "Dashboard Web": Python 3.12, pandas==2.0.0, plotly==5.18.0, dash==2.14.0

Sin entornos virtuales: IMPOSIBLE tener las 3 versiones de Python y Pandas.

Con entornos virtuales: Cada proyecto en su entorno, cero conflictos.

🚀 Ventaja de uv para Gestión de Versiones de Python:

Con venv tradicional, debes instalar manualmente cada versión de Python en tu sistema.

Con uv, es automático:

  • uv venv --python 3.8 → descarga e instala Python 3.8 automáticamente
  • uv venv --python 3.11 → descarga e instala Python 3.11 automáticamente
  • uv venv --python 3.12 → descarga e instala Python 3.12 automáticamente

¡No necesitas ir a python.org ni configurar nada manualmente!

✅ Regla de Oro del Desarrollo Python:

"NUNCA instalar paquetes en el Python del sistema. SIEMPRE usar un entorno virtual."

Esta práctica es obligatoria en desarrollo profesional y evita innumerables problemas.

3.2 Crear Entornos con venv (Método estándar, clásico y no recomendado)

venv es el módulo estándar de Python para crear entornos virtuales, incluido desde Python 3.3+.

Crear y activar entorno virtual

# Crear entorno virtual
python -m venv .venv

# Activar entorno (Windows PowerShell)
.venv\Scripts\Activate.ps1

# Activar entorno (macOS/Linux)
source .venv/bin/activate

# Una vez activado, instalar paquetes
pip install numpy pandas matplotlib

# Desactivar entorno
deactivate
⚠️ Política de Ejecución en Windows:

Si encuentras error al activar en PowerShell, ejecuta como administrador:

Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser

3.3 Crear Entornos con uv (Método moderno, rápido y recomendado)

uv simplifica la creación y gestión de entornos virtuales con comandos más directos y velocidad superior:

Creación Básica de Entornos con uv

⚠️ Importante: Si solo usas uv venv (sin uv init), NO puedes usar uv add. Debes usar uv pip install.

Entorno simple con uv venv (sin pyproject.toml)

# Crear entorno virtual simple
uv venv

# Crear con nombre personalizado
uv venv mi_entorno

# Crear con versión específica de Python
uv venv --python 3.12
uv venv --python 3.11

# Activar igual que con venv
# Windows: .venv\Scripts\Activate.ps1
# macOS/Linux: source .venv/bin/activate

# Instalar paquetes con uv pip (NO uses uv add aquí)
uv pip install numpy pandas matplotlib seaborn scikit-learn

# Desactivar
deactivate

uv init: Inicializar Proyectos Completos

uv init crea la estructura completa de un proyecto Python con pyproject.toml. Solo en proyectos inicializados con uv init puedes usar uv add para gestión declarativa de dependencias.

✅ Ventaja clave de uv init: Permite usar uv add para que las dependencias se registren automáticamente en pyproject.toml, facilitando la reproducibilidad.
📋 pyproject.toml vs requirements.txt:

pyproject.toml es el estándar moderno de Python (PEP 518, 2016) y la forma recomendada para proyectos nuevos:

  • Más moderno: Estándar oficial de Python desde 2016, reemplaza varios archivos antiguos (setup.py, setup.cfg, requirements.txt)
  • Gestión declarativa: Define dependencias principales, las herramientas calculan las transitivas automáticamente
  • Metadatos del proyecto: Incluye nombre, versión, autores, licencia, scripts, todo en un solo archivo
  • Compatibilidad con herramientas: Poetry, Hatch, PDM, y ahora uv usan pyproject.toml
  • Separación de dependencias: Distingue entre dependencias de producción y desarrollo

requirements.txt es el método tradicional, aún válido y ampliamente usado:

  • Simple y directo: Lista plana de paquetes con versiones exactas
  • Universalmente compatible: Funciona con pip, uv pip, virtualenv, etc.
  • Ideal para despliegues: Reproducción exacta de entornos en producción
  • Generado automáticamente: Se puede crear con pip freeze o uv pip freeze

🎯 Recomendación práctica:

  • Proyectos nuevos: Usa pyproject.toml con uv init + uv add
  • Proyectos legacy o simples: requirements.txt es perfectamente válido
  • Mejor de ambos: Puedes tener pyproject.toml para desarrollo y generar requirements.txt para despliegue
  • Colaboración: Si tu equipo ya usa uno, mantén la consistencia

Inicializar proyecto con uv init

# Crear nuevo proyecto con estructura completa
uv init mi_proyecto_data_science
cd mi_proyecto_data_science

# Estructura creada automáticamente:
# mi_proyecto_data_science/
# ├── pyproject.toml      # ✅ Permite usar uv add
# ├── README.md           # Documentación
# └── .python-version     # Versión de Python

# Crear entorno virtual (OPCIONAL si solo usarás uv add)
# uv add crea y gestiona el entorno automáticamente
uv venv

# Activar entorno (OPCIONAL si usarás uv add)
# Windows: .venv\Scripts\Activate.ps1
# macOS/Linux: source .venv/bin/activate

# ✅ AHORA SÍ puedes usar uv add (porque existe pyproject.toml)
# uv add creará el entorno automáticamente si no existe
uv add numpy pandas matplotlib

# Crear proyecto en directorio actual
uv init .

# Inicializar con plantilla específica
uv init --name "Análisis Ventas" --python 3.12 proyecto_ventas

uv run: Ejecutar Scripts sin Activar Entorno

uv run permite ejecutar scripts Python utilizando el entorno virtual sin necesidad de activarlo explícitamente.

Ejecutar código con uv run

# Ejecutar script usando el entorno virtual del proyecto
uv run python analisis.py

# Ejecutar módulo
uv run python -m pytest

# Ejecutar comando directo
uv run jupyter notebook

# Instalar y ejecutar en un solo paso
uv run --with pandas --with matplotlib python analisis.py

# Ventajas:
# - No necesitas activar el entorno
# - uv detecta automáticamente el .venv del proyecto
# - Ideal para CI/CD y automatización
# - Evita olvidar activar el entorno

uv sync: Sincronizar Dependencias

⚠️ REQUISITO OBLIGATORIO: uv sync SOLO funciona en proyectos con pyproject.toml (creados con uv init). Si solo usaste uv venv, este comando NO funcionará.

Sincronización de dependencias con uv

# ✅ Sincronizar entorno con pyproject.toml (requiere uv init)
uv sync

# Esto garantiza que el entorno tenga EXACTAMENTE
# las dependencias especificadas en pyproject.toml:
# - Instala las faltantes
# - Desinstala las que sobran
# - Actualiza las que tienen versión incorrecta

# Muy útil al clonar repositorios o cambiar de rama
git clone https://github.com/usuario/proyecto.git
cd proyecto
uv sync  # ¡Listo para trabajar!

# ❌ Si NO existe pyproject.toml:
# Error: No se encuentra pyproject.toml
# Solución: Usa requirements.txt con uv pip install -r requirements.txt

uv add: Agregar Dependencias al Proyecto

⚠️ REQUISITO OBLIGATORIO: uv add SOLO funciona si previamente has usado uv init (que crea pyproject.toml). Si solo creaste un entorno con uv venv, debes usar uv pip install.

Gestión declarativa con uv add (requiere uv init)

# ✅ Esto SOLO funciona después de uv init

# Agregar dependencia al proyecto (modifica pyproject.toml)
uv add pandas
uv add numpy>=1.24.0
uv add matplotlib seaborn plotly

# Agregar dependencia de desarrollo
uv add --dev pytest black flake8

# Esto actualiza automáticamente:
# - pyproject.toml (registro de dependencias)
# - Instala el paquete en el entorno
# - Genera/actualiza lockfile

# ❌ Si NO usaste uv init:
# Error: No se encuentra pyproject.toml
# Solución: Usa uv pip install en su lugar

  • Escrito en Rust: Rendimiento nativo y confiabilidad

3.4 Gestión de Dependencias con requirements.txt

⚠️ Nota sobre flujos modernos: Si usas uv init + uv add (recomendado para proyectos nuevos), las dependencias se gestionan automáticamente en pyproject.toml y NO necesitas crear ni gestionar manualmente requirements.txt. Este apartado es relevante para:
  • Proyectos legacy que ya usan requirements.txt
  • Flujo simple con uv venv + uv pip install (sin uv init)
  • Despliegue en producción (generar requirements.txt desde pyproject.toml)
  • Compatibilidad con herramientas que solo leen requirements.txt

El archivo requirements.txt documenta todas las dependencias del proyecto:

Trabajar con requirements.txt

# Generar requirements.txt
pip freeze > requirements.txt

# Instalar desde requirements.txt
pip install -r requirements.txt

# O con uv (más rápido)
uv pip install -r requirements.txt

Ejemplo de requirements.txt para Data Science

# requirements.txt
numpy>=1.24.0
pandas>=2.1.0
matplotlib>=3.7.0
seaborn>=0.12.0

3.5 Flujo de Trabajo Completo con venv y pip

⚠️ Flujo tradicional vs moderno: Este apartado muestra el flujo tradicional con venv y pip. Aunque sigue siendo válido, las prácticas modernas recomiendan usar uv en su lugar:
  • Flujo moderno recomendado: uv init + uv add (gestión automática con pyproject.toml)
  • Alternativa moderna simple: uv venv + uv pip install (10-100x más rápido que pip)
  • Cuándo usar venv + pip: Entornos sin uv instalado, scripts de CI/CD legacy, compatibilidad estricta
  • Ventaja de migrar a uv: Mayor velocidad, gestión automática de versiones de Python, mejor resolución de dependencias

Este contenido se mantiene por compatibilidad y comprensión del ecosistema tradicional.

Configuración de proyecto desde cero

pip install numpy pandas matplotlib seaborn scikit-learn jupyter

# 6. Generar requirements.txt
pip freeze > requirements.txt

# 7. Crear estructura del proyecto
mkdir data notebooks src

# 8. Iniciar trabajo
jupyter notebook
✅ Mejores prácticas con entornos virtuales (flujo moderno con uv):
  • Usar uv init para proyectos nuevos: Crea automáticamente .gitignore, pyproject.toml y estructura del proyecto
  • Gestión con uv add: Registra dependencias automáticamente en pyproject.toml
  • Un entorno por proyecto: Nunca compartir entornos entre proyectos diferentes
  • Nunca instalar en el sistema: Siempre usar entornos virtuales aislados
  • Versionar pyproject.toml: Incluir en Git para que el equipo use las mismas dependencias
  • .venv ya excluido: uv init crea .gitignore con .venv/ automáticamente

4.1 NumPy: La Fundación del Cálculo Numérico

NumPy (Numerical Python) es la base de todo el ecosistema científico de Python. Proporciona estructuras de datos eficientes para trabajar con arrays multidimensionales y operaciones matemáticas de alto rendimiento.

🔑 Conceptos clave de NumPy:
  • Arrays N-dimensionales: Estructuras optimizadas para datos numéricos
  • Vectorización: Operaciones sobre arrays completos sin bucles (hasta 100x más rápido)
  • Álgebra lineal: Multiplicación de matrices, descomposiciones, eigenvalores
  • Broadcasting: Operaciones entre arrays de diferentes formas

Ejemplo rápido de NumPy

import numpy as np

# Crear array
datos = np.array([1, 2, 3, 4, 5])

# Operaciones vectorizadas (sin bucles)
resultado = datos * 2 + 10  # [12, 14, 16, 18, 20]

# Operaciones estadísticas
media = datos.mean()  # 3.0
desviacion = datos.std()  # 1.41...

4.2 Pandas: El Rey del Análisis de Datos Tabulares

Pandas es la biblioteca fundamental para trabajar con datos estructurados (tablas). Proporciona estructuras de datos intuitivas y herramientas poderosas para análisis de datos.

🔑 Conceptos clave de Pandas:
  • DataFrames: Tablas bidimensionales con columnas etiquetadas
  • Series: Arrays unidimensionales etiquetados
  • Indexación flexible: Acceso por etiquetas o posiciones
  • Operaciones grupales: Agrupar, agregar y transformar datos
  • Gestión de datos faltantes: Detección y manejo de NaN

Ejemplo rápido de Pandas

import pandas as pd

# Crear DataFrame
ventas = pd.DataFrame({
    'producto': ['A', 'B', 'C', 'A', 'B'],
    'cantidad': [10, 15, 8, 12, 20],
    'precio': [100, 150, 80, 100, 150]
})

# Análisis rápido
total_por_producto = ventas.groupby('producto')['cantidad'].sum()
estadisticas = ventas.describe()

        return False

7. Trabajar con Jupyter Notebooks

Jupyter Notebooks son documentos interactivos que combinan código ejecutable, texto enriquecido, ecuaciones matemáticas, visualizaciones y medios multimedia. Son la herramienta estándar para análisis exploratorio de datos, prototipado rápido y comunicación de resultados.

7.1 ¿Qué es un Jupyter Notebook?

Un Jupyter Notebook (.ipynb) es un documento JSON que contiene una secuencia ordenada de celdas de entrada/salida. Cada celda puede contener:

📝 Tipos de celdas:
  • Celdas de código: Código Python ejecutable que produce resultados
  • Celdas Markdown: Texto formateado, títulos, listas, enlaces, imágenes, ecuaciones LaTeX
  • Celdas Raw: Texto sin procesar (raramente usado)

Estructura básica de un notebook

Un notebook típico de análisis de datos sigue esta estructura:

  1. Título y descripción (Markdown): Qué hace el notebook
  2. Importación de bibliotecas (Código): import pandas as pd, numpy as np
  3. Carga de datos (Código): Leer CSV, conexión a BD, etc.
  4. Exploración inicial (Código + Markdown): df.head(), df.info()
  5. Análisis y transformaciones (Código + Markdown): Limpieza, filtrado, agregación
  6. Visualizaciones (Código): Gráficos con matplotlib/seaborn
  7. Conclusiones (Markdown): Interpretación de resultados

7.2 Kernels y la Relación con ipykernel

Un kernel es el motor de ejecución que ejecuta el código de las celdas. Cuando ejecutas una celda de código en un notebook, el kernel:

🔧 Funcionamiento del kernel:
  1. Recibe el código de la celda
  2. Lo ejecuta en su espacio de memoria (sesión Python)
  3. Devuelve los resultados (texto, gráficos, errores)
  4. Mantiene el estado entre ejecuciones (variables, funciones, imports)
🔑 Concepto clave: Estado del kernel

El kernel mantiene un estado persistente durante toda la sesión:

  • Si defines x = 10 en la celda 1, x estará disponible en celdas posteriores
  • Puedes ejecutar celdas en cualquier orden, pero esto puede causar confusión
  • Reiniciar el kernel (Restart) borra todas las variables y vuelve al estado inicial
  • Buena práctica: Ejecutar "Restart & Run All" periódicamente para verificar que el notebook funciona de inicio a fin

¿Qué es ipykernel?

ipykernel es el paquete de Python que proporciona el kernel IPython para Jupyter. Es el puente entre el notebook y el intérprete Python.

📦 ipykernel y entornos virtuales:

⚠️ IMPORTANTE: Cada entorno virtual necesita tener ipykernel instalado para poder usarse como kernel en notebooks.

  • Sin ipykernel: El entorno virtual no aparecerá como opción de kernel
  • Instalación:uv add ipykernel o uv pip install ipykernel o pip install ipykernel
  • Verificar: Al seleccionar kernel en VS Code/PyCharm, debe aparecer tu entorno .venv

Configurar kernel de entorno virtual

# 1. Crear proyecto con uv
uv init mi_proyecto_notebooks --python 3.14
cd mi_proyecto_notebooks

# 2. Crear entorno virtual (opcional si luego voy a usar uv add)

# 3. Activar entorno (opcional, necesario sólo si voy a ejecutar scripts en la línea de comandos)
# Windows: .venv\Scripts\Activate.ps1
# macOS/Linux: source .venv/bin/activate

# 4. CRÍTICO: Instalar ipykernel y otras librerias
uv add ipykernel numpy pandas matplotlib seaborn


# 5. Verificar que el kernel está disponible
# Abrir VS Code y crear un notebook nuevo
code .
